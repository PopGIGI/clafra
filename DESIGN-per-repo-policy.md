# Per-repo policy scope

## Problem

Clafra's stated purpose is to optimize each project against its own SDLC — preventing context drift, surfacing stale assumptions, graduating patterns into tools when *that project's* signal warrants it. But the current implementation enforces a single global policy across every registered repo:

- **One cron line** (`17 3 * * *`) drives **one dispatcher** for all repos in `repos.json`.
- **One prompt template** (`build_prompt` in `scripts/cron-dispatch.sh:81`) is reused verbatim for poracle, ryokaniz, osint_agent, tft18.
- **One output schema** (Tier 1 mechanical / Tier 2 judgment) — no repo can declare its own focus areas or "definition of done."
- **One failure mode is global**: when `claude` was missing from cron PATH (2026-04-15 → today), all four repos went 22 days with zero analysis. There's no way to fix or pause one repo independently.

Per-repo `.clafra/` directories hold *artifacts* (patterns, criteria, pending reviews) but not *policy*. Policy lives upstream and is uniform.

This is the inverse of what an SDLC optimizer should be. A productivity tool ("get your shit done") works because it knows what *this* project considers shit getting done — and that varies wildly between a Polymarket trading bot and an OSINT scraper.

## Current state — concrete

| Layer | Where it lives | Per-repo? |
|---|---|---|
| Tool registry | `<repo>/.clafra/registry.json` | ✓ |
| Patterns | `<repo>/patterns.json` | ✓ |
| Success criteria | `<repo>/tools/*.json` `success_criteria` field | ✓ |
| Pending reviews | `<repo>/.clafra/pending-reviews/` | ✓ |
| **Cron schedule** | `crontab -l` (one line) | ✗ |
| **Dispatcher prompt** | `scripts/cron-dispatch.sh:81-127` | ✗ |
| **Tier definitions** | `scripts/cron-dispatch.sh` (hardcoded) | ✗ |
| **Repo selection** | `.clafra/repos.json` | ✗ (one list) |
| **Failure isolation** | none — one bad PATH kills all | ✗ |

## Proposed mechanism

Add a per-repo `<repo>/.clafra/policy.yaml` declaring how *that* project wants to be optimized. The dispatcher becomes a thin executor that reads each repo's policy rather than holding policy itself.

```yaml
# /home/pop/poracle/.clafra/policy.yaml
schedule: "17 3 * * *"          # cron expression, or "manual" / "on-commit"
focus_areas:
  - paper_bot_health             # custom check: data/no_paper/loop.log fresh, PID alive
  - pattern_graduation_signal    # standard: patterns at threshold ≥3
  - stale_tools                  # standard: criteria failing for N runs
  - documentation_drift          # custom: RESEARCH_PLAN.md vs config/*.yaml diff
tiers:
  mechanical:
    enabled: true
    auto_apply: false            # poracle prefers human gate even on mechanical
  judgment:
    enabled: true
    prompt_addendum: |
      This repo runs live trading. Flag any criterion change that could mask
      a regression in PnL accounting, balance reconciliation, or order placement.
definition_of_done:
  - "no stale tools"
  - "patterns flagged → reviewed within 1 milestone"
  - "open paper-bot positions reconciled with trade_history.jsonl"
notify:
  channel: discord
  webhook_env: DISCORD_CLAFRA_WEBHOOK
```

Each repo gets its own crontab line (or one dispatcher reads each policy's `schedule` and fans out). Failure isolation: a broken PATH or auth issue in one repo's run no longer blocks the others.

The dispatcher's prompt becomes a base template + the repo's `prompt_addendum` + the repo's `focus_areas` resolved against locally-defined check scripts in `.clafra/checks/`.

## Migration path

1. **Phase 1 — additive.** Add `policy.yaml` schema; if absent, dispatcher falls back to current global behavior. Zero breakage for existing repos.
2. **Phase 2 — first adopter.** poracle writes its policy.yaml. Validate the per-repo prompt and focus_areas pipeline end-to-end on one repo before generalizing.
3. **Phase 3 — split crontab.** Replace the single cron line with one per repo (generated by `clafra-init.sh`), each pointing at `cron-dispatch-one.sh <repo-path>`. Failure isolation lands here.
4. **Phase 4 — custom checks.** Allow repos to declare `focus_areas` that resolve to `.clafra/checks/*.sh` scripts the gather phase invokes. This is where SDLC specificity actually shows up.
5. **Phase 5 — retire `repos.json`** as the source of truth. Discovery becomes "any repo whose `.clafra/policy.yaml` opts in."

Phases 1–3 are mechanical. Phase 4 is the one that makes the framework actually project-aware; everything before it is plumbing.

## Open questions

- **Where do shared/global checks live?** Some criteria (e.g., "claude CLI present", "git remote reachable") are universal. Probably a `clafra/checks-stdlib/` that policies reference by name.
- **Should `policy.yaml` be checked into the project repo, or live in a separate per-repo overlay?** Checked-in is more transparent and survives clones; overlay is cleaner for personal preferences. Lean toward checked-in.
- **Auth caching across repos.** Today one `check_auth` call covers all repos in a single dispatcher run. With per-repo runs, do we re-auth N times? Probably acceptable — auth check is cheap.
- **Cross-repo learnings.** If poracle graduates a pattern that also appears in ryokaniz, should the framework surface that? This is a feature *enabled* by per-repo policy (each repo can declare its own threshold) but not solved by it. Out of scope for this note.
- **Backward compatibility for repos that never adopt policy.yaml.** Is "fall back to global" permanent, or do we deprecate it after a grace period? Recommendation: permanent fallback, since some repos genuinely want zero customization.

## Why this matters

The 22-day silent cron failure is a symptom, not the disease. The disease is that clafra has no concept of project scope: it can't be partially broken, partially configured, or partially adopted. That makes it brittle (one PATH bug kills everything) and generic (every repo gets the same review prompt regardless of what "good" looks like there).

Per-repo policy is what turns clafra from "uniform governance layer" into "SDLC optimizer that respects each project's shape."
